CI Pipeline

GitHub Actions workflow — .github/workflows/ci.yml

⚡ git push
on: push — fires on every push to any branch
↓
🔵 Job 1 — Lint no dependencies
1
Checkout code
actions/checkout@v6 — downloads your repo onto the fresh Ubuntu VM. Without this the runner has no files to work with.
2
Setup Node.js
actions/setup-node@v5 with node-version: 22 and cache: npm — installs Node 22 and caches node_modules to speed up future runs.
3
Install dependencies
npm ci — clean, deterministic install from package-lock.json. Preferred over npm install in CI environments.
4
Run linter
npm run lint — runs ESLint on all .jsx files in src/. Any lint error fails this job and stops the pipeline.
❌ lint fails → pipeline stops here
↓ passes
🟢 Job 2 — Test needs: lint
1
Checkout code
Fresh VM — repo must be checked out again with actions/checkout@v6.
2
Setup Node.js
actions/setup-node@v5 — Node 22 installed again on this new VM.
3
Install dependencies
npm ci — clean install again on this VM.
4
Run tests
npm run test — runs Vitest. Any failing test stops the pipeline and deploy never runs.
❌ test fails → pipeline stops here
↓ passes
🟡 Job 3 — Deploy needs: test
1
Checkout code
Fresh VM again — actions/checkout@v6 checks out the repo.
2
Setup Node.js
actions/setup-node@v5 — Node 22 on this VM.
3
Install dependencies
npm ci — clean install.
4
Build project
npm run build — runs Vite to produce the production bundle in dist/.
5
Deploy
Placeholder step — replace with your real deployment command (e.g. Netlify, Vercel, AWS S3, etc.).

💡 Key concept — each job runs on a fresh VM.
GitHub spins up a brand-new ubuntu-latest virtual machine for every job. That's why checkout, Node setup, and npm ci are repeated in each job — nothing carries over between them.

💡 needs: controls the order.
Without needs:, all jobs run in parallel. Adding needs: lint to the test job and needs: test to deploy creates the sequential gate pattern: bad code can never reach deployment.
